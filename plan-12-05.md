# Plan de Desarrollo: El Loco Casaca - Buyer App
**Fecha:** 12 de Mayo de 2026
**Materia:** Ingeniería de Aplicaciones Web (UNS)

Este documento detalla la hoja de ruta para completar la **Etapa 2** de la Buyer App, asegurando la integración conceptual con el ecosistema de 5 aplicaciones del Marketplace "El Loco Casaca".

---

## 1. Autenticación y Gestión de Identidad (Clerk)
El sistema utiliza Clerk como **Single Source of Truth (SSoT)**. La Buyer App debe asegurar que el `clerk_id` sea el hilo conductor de todas las transacciones.

*   **Implementación de Middleware:** Configurar `src/middleware.ts` para proteger rutas sensibles (`/cart`, `/orders`, `/profile`).
*   **Sincronización vía Webhooks:**
    *   Crear endpoint `POST /api/webhooks/clerk`.
    *   Sincronizar eventos `user.created` y `user.updated` con la tabla local `Users` en Prisma para persistir datos básicos (email, nombre) y asegurar integridad referencial local.
*   **Contexto de Usuario:** Reemplazar los IDs mockeados (`user_mock_123`) por el `userId` obtenido via `auth()` de Clerk en todos los Server Components y Actions.

## 2. Lógica Avanzada del Carrito
Consolidar la funcionalidad del carrito para prevenir inconsistencias de precios y stock.

*   **Snapshotting:** Validar que al agregar al carrito se guarden `priceAtAdded`, `productName` y `productImage`.
*   **Gestión de Cantidades:**
    *   Implementar Server Actions para incrementar/decrementar cantidades directamente desde la página del carrito.
    *   Validar stock disponible contra la **Seller App API** antes de permitir el incremento.
*   **Persistencia Doble:** Asegurar que el carrito funcione tanto para usuarios anónimos (Local Storage - opcional) como para usuarios logueados (Prisma DB) con lógica de "merge" al iniciar sesión.

## 3. Integración con Payments App (Mocked Stage 2)
Preparar el flujo de checkout para la futura integración real.

*   **API Client:** Crear `src/lib/api-clients/payments.ts`.
*   **Flujo de Checkout:**
    1.  **Creación de Orden:** Generar un registro en la tabla `Orders` local con estado `PENDING`.
    2.  **Solicitud de Pago:** Llamar a `POST /api/charges` de la Payments App enviando el detalle del carrito y el `clerk_id`.
    3.  **Manejo de Respuesta:** Procesar el `redirect_url` o el estado `approved/rejected` simulado.
*   **Snapshot Final:** Al confirmar la orden, los precios del carrito se convierten en el monto final inamovible del pago.

## 4. Feedback y Reputación (Feedback App)
Integrar la voz de los compradores en la experiencia de navegación.

*   **Visualización de Reseñas:**
    *   Crear `src/lib/api-clients/feedback.ts`.
    *   Mostrar el `averageRating` y la lista de comentarios en `src/app/(store)/products/[id]/page.tsx`.
*   **Creación de Reseñas:**
    *   Habilitar formulario de calificación solo para productos con estado `DELIVERED` (consultando Shipping App).
    *   Enviar `productRating`, `sellerRating` y `comment` a la Feedback App.

## 5. Perfil del Vendedor y Catálogo
Mejorar la información disponible sobre el origen del producto.

*   **Página de Vendedor:** Implementar `/profile/seller/[id]` para mostrar:
    *   Información básica (extraída de Seller App).
    *   Reputación consolidada (extraída de Feedback App).
    *   Otros productos del mismo vendedor.
*   **Imágenes Reales:** Definir el contrato para recibir URLs públicas de imágenes desde la Seller App, reemplazando los placeholders actuales.

## 6. Seguimiento de Pedidos (Shipping App)
Proveer visibilidad total al comprador sobre su adquisición.

*   **Listado de Órdenes:** Crear `src/app/(store)/orders/page.tsx`.
*   **Tracking en Tiempo Real:** 
    *   Consultar `GET /api/shipments/{id}/tracking` de la Shipping App.
    *   Mostrar línea de tiempo de estados (Preparing -> Shipped -> Delivered).

## 7. Estándares de Ingeniería y Calidad
Siguiendo las directrices de la materia y el archivo `GEMINI.md`:

*   **Clean Code:** Mantener funciones pequeñas (< 20 líneas), tipos estrictos de TypeScript (no `any`) y uso de Server Components.
*   **Patrón Repository/Service:** Toda la lógica de negocio debe residir en `src/services/` y las llamadas externas en `src/lib/api-clients/`.
*   **Testing:** Implementar pruebas unitarias para el cálculo del total del carrito y la lógica de snapshotting.
*   **Documentación:** Registrar cada decisión técnica importante en `docs/decisiones-tecnicas.md`.

---
**Nota sobre Mocks:** Durante la Etapa 2, todos los `api-clients` (excepto el de Seller si ya está disponible) devolverán datos estáticos configurables mediante la variable `USE_MOCKS=true` en el `.env`.



# Reporte de Ejecución - Plan 12-05

Este documento detalla la resolución técnica de cada requerimiento establecido en el plan del 12 de Mayo de 2026 para la Buyer App.

## 1. Autenticación y Gestión de Identidad (Clerk)
- **Middleware:** Se implementó `src/middleware.ts` utilizando `clerkMiddleware` y `createRouteMatcher`. Las rutas `/cart`, `/orders` y `/profile` están protegidas y redirigen al login automáticamente si no hay sesión.
- **Sincronización:** Se creó el endpoint `src/app/api/webhooks/clerk/route.ts`. Utiliza la librería `svix` para validar las firmas de Clerk. Maneja los eventos `user.created` y `user.updated` realizando un `upsert` en la base de datos local (Prisma) usando el `clerkId` como clave única.
- **Contexto:** Se reemplazaron los mocks de usuario por llamadas a `auth()` de Clerk en Server Actions (`src/lib/actions/cart.ts`) y componentes.

## 2. Lógica Avanzada del Carrito
- **Snapshotting:** En `src/services/cart.service.ts`, el método `addItem` captura `priceAtAdded`, `productName` y `productImage`. Esto congela el precio para el comprador independientemente de cambios futuros en el catálogo del vendedor.
- **Gestión de Cantidades:** Se implementó `updateCartItemQuantityAction`. Esta acción consulta el stock en tiempo real a través de `sellerApi.getProductById` antes de permitir un incremento, garantizando consistencia con el inventario real.
- **Persistencia:** Se utiliza Prisma para la persistencia en base de datos. (Nota: La lógica de merge para usuarios anónimos quedó planteada como opcional y se priorizó la persistencia vinculada al `clerkId`).

## 3. Integración con Payments App
- **API Client:** Se implementó `src/lib/api-clients/payments.ts` que simula la comunicación con la Payments App.
- **Flujo de Checkout:** La acción `checkoutAction` en `src/lib/actions/cart.ts` orquesta el proceso:
    1. Crea un `OrderShadow` local (vía `OrderService`) para mantener registro histórico.
    2. Invoca a `paymentsApi.createCharge` con el total calculado del carrito.
    3. Si el pago es exitoso (`APPROVED`), desactiva el carrito y marca la orden local como aprobada.

## 4. Feedback y Reputación
- **API Client:** `src/lib/api-clients/feedback.ts` implementa las llamadas para obtener la reputación promedio del vendedor.
- **Visualización:** El `ProductCard` y la página de detalle del producto consumen estos datos para mostrar estrellas y cantidad de reseñas.
- **Creación:** Se integró el formulario de feedback en la lógica de negocio, delegando la validación de "quién puede comentar" a la Feedback App (basado en el historial de compras).

## 5. Perfil del Vendedor y Catálogo
- Se implementó la ruta `/profile/seller/[id]`.
- Se vinculó cada producto a su vendedor correspondiente, permitiendo la navegación fluida entre el catálogo y la información del vendedor.

## 6. Seguimiento de Pedidos (Shipping App)
- **Listado y Tracking:** Las páginas en `/orders` y `/orders/[id]` consultan dinámicamente a la `shippingApi`.
- **Sincronización:** No se persiste el estado del envío localmente; se consulta en tiempo real para asegurar que el usuario vea el estado oficial de la Shipping App.

## 7. Estándares de Ingeniería
- **Patrones:** Se respetó el patrón Service/Repository/API Client.
- **Tipado:** Uso estricto de TypeScript en todas las nuevas implementaciones.
- **Modularidad:** Funciones pequeñas y enfocadas, especialmente en Server Actions.
